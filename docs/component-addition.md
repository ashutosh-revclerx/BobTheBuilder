# Adding New Components to BobTheBuilder

A comprehensive guide to extending the dashboard builder with new component types.

## 1. Introduction

### What is a Component?

In BobTheBuilder, a **component** is a reusable, configuration-driven UI element that appears on a dashboard. Components are not hardcoded pages or static designs — they are dynamically rendered based on a configuration object.

The core principle is:

```
UI = function(ComponentConfig)
```

This means:
- The **same React component** can render 100 different dashboards
- **Configuration objects** (JSON) drive all visual and behavioral variations
- **No code changes** are required to modify a component's appearance or behavior
- **Components are serializable** — they can be exported, imported, versioned, and regenerated by AI

### Configuration-Driven Architecture

Every component receives:
1. **Static Props** — what data to display (from queries or mock data)
2. **Style Object** — colors, spacing, fonts, borders
3. **Data Binding** — which database fields map to which properties
4. **Visibility Rules** — role-based conditional rendering
5. **Behavior Settings** — responsive behavior, interaction rules

Example:

```javascript
{
  type: "StatCard",
  id: "card-1",
  style: {
    backgroundColor: "#ffffff",
    textColor: "#0f1117",
    borderColor: "#e5e7eb"
  },
  data: {
    title: "Revenue",
    metric: "{{queries.salesQuery.total}}",
    trend: "+12%"
  },
  layout: { x: 0, y: 0, w: 4, h: 6 }
}
```

### Why Configuration Matters

Configuration enables:
- **Serialization**: Export/import dashboards as JSON
- **Consistency**: Same component renders identically in preview and final dashboard
- **AI Generation**: Gemini can generate valid configurations without coding
- **Theming**: Apply design systems globally without touching component code
- **Version Control**: Track dashboard changes in git
- **Reusability**: Share components across projects

## 4. Step-by-Step: Adding ProgressCard Component

Let's walk through adding a hypothetical **ProgressCard** component that displays a metric with a progress bar.

### Example: ProgressCard Component

A ProgressCard shows:
- A title ("Completion Rate")
- A metric value with unit ("85%")
- A visual progress bar (0-100%)
- Optional trend indicator ("+5% this week")

### Step 1: Create the Component UI

Create: `apps/frontend/src/components/dashboard-components/ProgressCard.tsx`

```typescript
import React from 'react';
import type { ComponentType } from '../../types/template';

interface ProgressCardProps {
  config: {
    type: ComponentType;
    style?: {
      backgroundColor?: string;
      textColor?: string;
      borderColor?: string;
      borderWidth?: number;
      borderRadius?: number;
      padding?: number;
    };
    data?: {
      title?: string;
      value?: number;
      unit?: string;
      trend?: string;
      trendValue?: string;
    };
  };
  readOnly?: boolean;
}

export default function ProgressCard({ config, readOnly = false }: ProgressCardProps) {
  const {
    title = 'Progress',
    value = 75,
    unit = '%',
    trend = '',
    trendValue = '',
  } = config.data || {};

  const style = config.style || {};
  const bgColor = style.backgroundColor || '#ffffff';
  const textColor = style.textColor || '#0f1117';
  const borderColor = style.borderColor || '#e5e7eb';
  const borderWidth = style.borderWidth || 1;
  const borderRadius = style.borderRadius || 8;
  const padding = style.padding || 16;

  // Ensure value is between 0 and 100
  const normalizedValue = Math.min(100, Math.max(0, value || 0));

  return (
    <div
      style={{
        backgroundColor: bgColor,
        color: textColor,
        border: `${borderWidth}px solid ${borderColor}`,
        borderRadius: `${borderRadius}px`,
        padding: `${padding}px`,
        display: 'flex',
        flexDirection: 'column',
        gap: '12px',
        height: '100%',
      }}
    >
      {/* Header */}
      <div style={{ display: 'flex', justifyContent: 'space-between', alignItems: 'flex-start' }}>
        <h3 style={{ margin: 0, fontSize: '14px', fontWeight: 500 }}>
          {title}
        </h3>
        {trend && (
          <span style={{ fontSize: '12px', color: '#6366f1', fontWeight: 600 }}>
            {trend}
          </span>
        )}
      </div>

      {/* Metric Value */}
      <div style={{ display: 'flex', alignItems: 'baseline', gap: '4px' }}>
        <span style={{ fontSize: '28px', fontWeight: 700 }}>
          {value}
        </span>
        {unit && (
          <span style={{ fontSize: '14px', color: 'currentColor', opacity: 0.7 }}>
            {unit}
          </span>
        )}
      </div>

      {/* Progress Bar */}
      <div
        style={{
          height: '8px',
          backgroundColor: 'rgba(0, 0, 0, 0.1)',
          borderRadius: '4px',
          overflow: 'hidden',
        }}
      >
        <div
          style={{
            height: '100%',
            width: `${normalizedValue}%`,
            backgroundColor: '#6366f1',
            transition: 'width 0.3s ease',
          }}
        />
      </div>

      {/* Trend Details */}
      {trendValue && (
        <div style={{ fontSize: '12px', color: 'rgba(0, 0, 0, 0.6)' }}>
          {trendValue}
        </div>
      )}
    </div>
  );
}
```

**Key Principles:**
- ✅ Config-driven (all data comes from `config` prop)
- ✅ No API calls (accepts data as props)
- ✅ Supports styling customization
- ✅ Responsive layout (uses flexbox)
- ✅ Theme-aware (uses style colors from config)
- ✅ Optional `readOnly` prop for preview mode

### Step 9: AI Generation Compatibility

The AI system needs to know about your component through **two independent paths**:

1. **Dashboard generation** (`POST /generate`) — uses `component_capabilities.py` injected into the Gemini system prompt
2. **Editor chat assistant** (`POST /chat`) — uses `component_capabilities.py` in every chat request to understand component properties

Both paths read from a single source-of-truth file: `component_capabilities.py`. If you don't register your component there, **neither the AI generator nor the editor assistant will know it exists**.

#### Part A — Register in `component_capabilities.py` (REQUIRED)

Edit: `apps/backend/services/llm/app/component_capabilities.py`

Add your component to the `COMPONENT_CAPABILITIES` dictionary:

```python
COMPONENT_CAPABILITIES: dict[str, dict] = {
    # ... existing components ...
    
    "ProgressCard": {
        "description": "Displays a metric with a visual progress bar. Ideal for showing completion rates, progress towards goals, or percentages.",
        "visual_role": "KPI card with animated progress indicator.",
        "style": COMMON_CARD_STYLE + [
            "progressColor",        # hex color for progress bar fill
            "trackColor",           # hex color for progress bar background
        ],
        "data": COMMON_VISIBILITY_DATA + [
            "title",                # string label
            "value",                # number 0-100
            "unit",                 # string suffix (e.g. "%", "pts")
            "trend",                # string (e.g. "+5%")
            "trendValue",           # string details (e.g. "Up from last week")
        ],
        "required": {
            "style": [],            # none are strictly required
            "data": ["title", "value"],  # these properties must always be present
        },
    },
}
```

**Key fields:**
- `description` — brief use case; appears in both generation and chat assistant prompts
- `visual_role` — tells Gemini where/when to place the component (e.g., "KPI card", "Data explorer")
- `style` — list of style property names the component accepts; only these are valid in generated configs
- `data` — list of data property names the component accepts; only these are valid in generated configs
- `required.style` / `required.data` — fields that must be present in every generated instance; prevents silent failures from missing required props

**How `component_capabilities.py` is consumed:**

1. **Generation flow** (`prompts.py::build_system_prompt()`):
   - Calls `format_capabilities_for_prompt()` to render the entire `COMPONENT_CAPABILITIES` registry as a large structured text block
   - Injects this block into the Gemini system prompt before every dashboard generation
   - Gemini uses this to know what components exist and what properties to generate for each

2. **Chat assistant flow** (`chat.py::build_chat_system_prompt()`):
   - Every chat request calls `_component_capabilities_summary()` which iterates `COMPONENT_CAPABILITIES`
   - Builds a compact reference: `- ProgressCard: style[backgroundColor, ...] data[title, value, ...]`
   - Injects into the chat system prompt so the assistant knows valid properties for your component
   - Also calls `COMPONENT_CAPABILITIES.keys()` to enumerate valid component type names

**Critical rule:** Property names in `COMPONENT_CAPABILITIES` must exactly match what your React component reads from `config.style` and `config.data`. Mismatched names cause the component to silently ignore AI-generated properties (see Common Mistakes below).

#### Part B — Register in `tools.py::PROPERTY_HINTS` (OPTIONAL but RECOMMENDED)

During generation, Gemini can call the `get_component_capabilities` tool to ask for detailed property hints for a specific component. These rich descriptions help Gemini make better decisions.

#### Part C — Add Pydantic schema to `schemas.py` (OPTIONAL but RECOMMENDED)

For request validation on the backend, add a Pydantic model for your component.

Edit: `apps/backend/services/llm/app/schemas.py`

```python
class ProgressCardComponent(BaseComponent):
    """Progress indicator with visual bar and trend indicator."""
    type: Literal["ProgressCard"] = "ProgressCard"
    
    class Data(BaseModel):
        title: Optional[str] = "Progress"
        value: Optional[int] = 75
        unit: Optional[str] = "%"
        trend: Optional[str] = None
        trendValue: Optional[str] = None
    
    class Style(BaseModel):
        backgroundColor: Optional[str] = None
        textColor: Optional[str] = None
        borderColor: Optional[str] = None
        borderWidth: Optional[int] = 1
        borderRadius: Optional[int] = 8
        padding: Optional[int] = 16
        progressColor: Optional[str] = "#6366f1"
        trackColor: Optional[str] = "#e5e7eb"
    
    style: Optional[Style] = Field(default_factory=Style)
    data: Data = Field(default_factory=Data)
```

This validates incoming API requests and prevents invalid data from reaching the frontend.

**Why this three-part process matters:**
- ✅ **Part A (required):** Both AI generation and editor chat immediately know your component exists
- ✅ **Part B (optional):** Gemini makes better property decisions during generation
- ✅ **Part C (optional):** Backend validates data before sending to frontend; prevents corruption
- ✅ Property names are centralized — change them in one place, both AI flows automatically update

## 5. Best Practices

### ✅ Keep Components Config-Driven

```typescript
// GOOD: All behavior from config
export default function ProgressCard({ config }: Props) {
  const { title, value } = config.data || {};
  return <div>{title}: {value}%</div>;
}

// BAD: Hardcoded behavior
export default function ProgressCard() {
  const value = 75;  // Hardcoded!
  return <div>Progress: {value}%</div>;
}
```

### ✅ Avoid Tightly Coupled Logic

```typescript
// GOOD: Accepts data, no external dependencies
function ProgressCard({ config }: Props) {
  return <div>{config.data.value}</div>;
}

// BAD: Fetches its own data
function ProgressCard() {
  const [data, setData] = useState(null);
  useEffect(() => {
    fetch('/api/progress').then(setData);  // Tightly coupled!
  }, []);
  return <div>{data?.value}</div>;
}
```

**Why?** Data fetching is handled by the query engine. Components should remain presentational.

### ✅ Support Theme Tokens

```typescript
// GOOD: Uses theme values
const bgColor = style.backgroundColor || theme.colors.panel;

// BAD: Hardcoded colors
const bgColor = style.backgroundColor || '#ffffff';
```

### ✅ Ensure Serialization Works

```typescript
// GOOD: Plain data structures
const config = {
  data: {
    title: 'Progress',
    value: 85,
  },
};
JSON.stringify(config);  // Works perfectly

// BAD: Non-serializable values
const config = {
  data: {
    title: 'Progress',
    fetchFn: () => fetch('/api'),  // Can't serialize functions!
  },
};
JSON.stringify(config);  // Error
```

### ✅ Support Mobile Behavior

```typescript
// GOOD: Responsive layout
<div style={{ display: 'flex', flexDirection: 'column', width: '100%' }}>
  {/* Stacks naturally on mobile */}
</div>

// BAD: Fixed dimensions
<div style={{ width: '500px', height: '400px' }}>
  {/* Breaks on mobile */}
</div>
```

### ✅ Include Fallback Values

```typescript
// GOOD: Safe defaults
const title = config.data?.title || 'Default Title';
const value = config.data?.value ?? 0;

// BAD: Crashes if missing
const title = config.data.title;  // TypeError if undefined!
```

## 6. Common Mistakes

### ❌ Hardcoded Styling

```typescript
// WRONG
<div style={{ backgroundColor: '#ffffff', color: '#0f1117' }}>
  {/* Colors never change, theme ignored */}
</div>

// CORRECT
<div style={{ backgroundColor: style.backgroundColor || theme.panel }}>
  {/* Respects configuration and theme */}
</div>
```

**Consequence:** Components look wrong when themes change. Impossible to customize. Breaks design system.

---

### ❌ Missing Schema Registration

```typescript
// WRONG: Forgot to add to TypeScript union
export type ComponentType = 'StatCard' | 'Table' | 'Button';
// ProgressCard is NOT in the union!

// CORRECT
export type ComponentType = 'StatCard' | 'Table' | 'Button' | 'ProgressCard';
```

**Consequence:** Type errors. IDE autocomplete broken. Runtime errors when component is selected.

---

### ❌ Not Registering in RenderRegistry

```typescript
// WRONG: Component exists but not in registry
export const RenderRegistry = {
  StatCard,
  Table,
  Button,
  // ProgressCard is missing!
};

// CORRECT
export const RenderRegistry = {
  StatCard,
  Table,
  Button,
  ProgressCard,  // Registered
};
```

**Consequence:** Component won't render. Canvas shows `unknown component`. Export fails. Preview breaks.

---

### ❌ Property Panel Inconsistency

```typescript
// WRONG: Each component has different structure
if (type === 'StatCard') {
  return <StatCardPanel />;  // Different implementation
}
if (type === 'ProgressCard') {
  return <ProgressCardPanel />;  // Different structure
}

// CORRECT: Use consistent patterns
{type === 'ProgressCard' && (
  <div className="properties-section">
    <PropertyInput label="Title" value={...} onChange={...} />
    <PropertyColorPicker label="Color" value={...} onChange={...} />
  </div>
)}
```

**Consequence:** UI inconsistency. Confusing UX. Harder to maintain.

---

### ❌ Missing Mobile Support

```typescript
// WRONG: Assumes desktop size
<div style={{ width: '500px', height: '300px' }}>

// CORRECT: Responsive
<div style={{ width: '100%', height: 'auto', minHeight: '200px' }}>
```

**Consequence:** Broken on mobile. Overflow issues. Poor user experience. Lower adoption.

---

### ❌ Export Serialization Issues

```typescript
// WRONG: Non-serializable data
const config = {
  data: {
    formatter: (val) => val.toFixed(2),  // Functions don't serialize!
  },
};

// CORRECT: Serializable values only
const config = {
  data: {
    decimals: 2,  // Let component handle formatting
  },
};
```

**Consequence:** Export fails silently. Import corrupts data. Users lose work.

---

### ❌ Duplicate Rendering Logic

```typescript
// WRONG: Copy-pasting style rendering
// In Canvas.tsx
const renderProgressCard = (c) => {
  const bgColor = c.style.backgroundColor || '#fff';
  // ... rendering logic

// In PreviewRenderer.tsx
const renderProgressCard = (c) => {
  const bgColor = c.style.backgroundColor || '#fff';
  // ... same logic duplicated!

// CORRECT: Use RenderRegistry (single source of truth)
// Both Canvas and PreviewRenderer import from RenderRegistry
const Component = RenderRegistry['ProgressCard'];
return <Component config={c} />;
```

**Consequence:** Preview doesn't match final dashboard. Inconsistent rendering. Visual divergence. AI-generated dashboards look different.

---

### ❌ Wrong Property Names in `component_capabilities.py`

```python
# WRONG: Property names don't match React component
"ProgressCard": {
    "style": COMMON_CARD_STYLE + [
        "barColor",        # Component reads "progressColor" — mismatch!
        "backgroundColor",
    ],
    "data": COMMON_VISIBILITY_DATA + [
        "percent",         # Component reads "value" — mismatch!
        "progressValue",   # Wrong name entirely
    ],
}

// React component reads:
const { value } = config.data;
const { progressColor } = config.style;
// These don't exist in the AI-generated config!

// CORRECT: Names match exactly
"ProgressCard": {
    "style": COMMON_CARD_STYLE + [
        "progressColor",   # Matches config.style.progressColor
        "trackColor",
    ],
    "data": COMMON_VISIBILITY_DATA + [
        "value",           # Matches config.data.value
        "unit",
        "trend",
    ],
}
```

**Consequence:** AI generates valid-looking configs with properties that don't exist. React component silently renders with defaults (fallback values), ignoring all AI customization. Dashboard looks generic and wrong. The component appears broken, but no error is logged. Very hard to debug.

## 10. Getting Help

If you get stuck:

1. **Component won't render?** → Check `RenderRegistry`
2. **Types are wrong?** → Check `ComponentType` union and interfaces
3. **Properties don't update?** → Check property panel onChange handlers
4. **Export fails?** → Check `exportService.ts` and serialization
5. **AI doesn't generate it?** → Check LLM prompts and schemas
6. **Looks wrong in preview?** → Check `PreviewRenderer` and `resolveBackground()`
7. **Mobile looks broken?** → Check responsive styles (no fixed widths)

Consult existing components (StatCard, Table, Button) as reference implementations.
